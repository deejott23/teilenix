01  ·  Technologiestrategie

Welcher Weg zur nativen App?

Drei realisierbare Ansätze, die die vorhandene Next.js + Supabase Infrastruktur unterschiedlich stark nutzen. Bewertet nach Aufwand, Code-Wiederverwendung und Ergebnis.

⭐ Empfohlen
Capacitor
Bestehenden Next.js Code in eine native App verpacken — minimaler Umbau, maximale Geschwindigkeit.
Aufwand: Gering Code: Minimal
⏱ 2–4 Wochen
  • ✓95% Code-Wiederverwendung
  • ✓Schnell & kostengünstig
  • ✓Native Plugins verfügbar
  • ✗Kein vollständig nativer Look
  • ✗WebView-Basis (Performance)

React Native / Expo

02  ·  Empfohlener Weg: Capacitor

4-Wochen Roadmap

Schritt-für-Schritt Plan mit Capacitor als Wrapper. Die Web-App läuft in einem nativen WebView-Container mit Zugang zu nativen iOS- und Android-APIs.

Phase 1 · Woche 1
Setup & Konfiguration
  • Capacitor CLI installieren und in Next.js Projekt integrieren
  • Xcode (Mac) und Android Studio einrichten
  • Native Projekte generieren: npx cap add ios / npx cap add android
  • Vercel API-URL als Basis-Endpunkt konfigurieren
Phase 2 · Woche 2
Native Features & Assets
  • Push Notifications via Capacitor Plugin (APNs für iOS, FCM für Android)
  • Haptic Feedback und Share-Dialog einbinden
  • App Icon (1024×1024px) für beide Plattformen generieren
  • Splashscreen Design und Animation konfigurieren
Phase 3 · Woche 3
Build, Test & Bugfixing
  • iOS Build in Xcode erstellen und auf Simulator testen
  • Android Build in Android Studio und auf Emulator testen
  • Deep Links für /join/[code] konfigurieren
  • Auth-Flow mit Token-Persistenz testen (kein Cookie-Zugriff in WebView)
Phase 4 · Woche 4
Store Submission & Review
  • App Store Connect: Metadaten, Screenshots, Datenschutz-Informationen
  • TestFlight Beta-Test mit echten Nutzern (iOS)
  • Google Play Internal Testing Track einrichten (Android)
  • App Store Submission → Review-Prozess abwarten (1–7 Tage)

05  ·  Kritische Herausforderung

Das größte technische Problem

⚠️ Next.js Server Components in nativer App nicht möglich
Next.js Server Components und API Routes erfordern einen laufenden Node.js-Server. Eine native App (iOS/Android) hat keinen Server — der Code läuft ausschließlich auf dem Gerät des Nutzers.

Das bedeutet: Alle async Seiten-Komponenten, Server Actions und API Route Handlers können in der nativen App nicht direkt ausgeführt werden.
Empfohlene Lösung: API auf Vercel belassen. Die native App kommuniziert per HTTPS mit den bestehenden Vercel API-Routes. Dadurch funktionieren ~95% des Codes ohne Änderung — nur die Auth-Session-Verwaltung und das statische Rendering müssen angepasst werden.
06  ·  Empfehlung & Nächste Schritte

Wie jetzt vorgehen?

⭐ Empfehlung
Capacitor + Vercel API — der Hybrid-Ansatz
Starte mit Capacitor als WebView-Wrapper und behalte die Vercel API als Backend. Minimaler Aufwand, maximale Code-Wiederverwendung. Die App läuft als WebView mit nativen Erweiterungen (Push, Haptics, Share) — und landet trotzdem im App Store.
