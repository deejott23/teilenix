05 — Kritische Herausforderung

Das größte Problem

Ein technisches Detail, das frühzeitig bedacht werden muss.

⚠ Next.js Server Components in nativer App
Next.js Server Components und API Routes benötigen einen laufenden Node.js-Server. Eine native App kann keinen Server starten — sie ist eine statische Datei. Das bedeutet: Server-seitige Logik funktioniert nicht direkt in der App.
✓ Empfohlene Lösung: Hybrid-Ansatz
Die App kommuniziert via HTTPS mit den bestehenden API Routes auf Vercel. Die Next.js App bleibt auf Vercel deployed — die native App ist nur ein WebView-Wrapper, der diese URL lädt. Damit funktionieren 95% des bestehenden Codes ohne jede Änderung. Server Components, API Routes, Auth — alles bleibt wie gehabt.
06 — Offline-Fähigkeit

Ausgaben ohne Internet erfassen

Mit Capacitor ist echte Offline-Unterstützung machbar — Ausgaben lokal speichern, bei Verbindung automatisch synchronisieren.

✓ Mit Capacitor grundsätzlich umsetzbar
Capacitor bietet Zugang zu nativen Storage-APIs und SQLite direkt auf dem Gerät. Damit können Ausgaben lokal gespeichert und später mit Supabase synchronisiert werden — auch wenn Tage vergehen.
Wie es funktioniert
→  Neue Ausgabe wird zuerst lokal in SQLite oder Capacitor Preferences gespeichert
→  App erkennt automatisch ob Online- oder Offline-Status
→  Bei Verbindungsrückkehr: automatischer Sync zu Supabase
→  Konfliktlösung: Timestamp-basiert (neuere Einträge gewinnen)
→  Offline-Badge in der App zeigt Sync-Status
Benötigte Pakete
@capacitor-community/sqlite Kostenlos
@capacitor/network Kostenlos
Supabase Realtime Inklusive
PowerSync (Alternative) 0–50 USD/Monat
✓ Offline möglich
✓  Ausgaben erfassen & speichern
✓  Eigene Ausgaben ansehen
✓  Zuletzt geladene Ausgaben lesen
✓  Kategorien & Teilnehmer (gecacht)
✗ Nur mit Internet

Aufwand-Einschätzung
Einfach: Eingabe lokal puffern, bei Online-Rückkehr senden +1–2 Wochen
Mittel: Lokale SQLite-Datenbank mit vollständigem Sync +3–5 Wochen
PowerSync (fertige Offline-First Sync-Lösung für Supabase) +1–2 Wochen
⚠ Empfehlung: Einfache Variante zuerst
Start mit der einfachen Variante: Formular-Eingaben lokal im Gerätespeicher puffern (Capacitor Preferences), bei Verbindungsrückkehr automatisch absenden. Das deckt 90% des Offline-Bedarfs mit minimalem Aufwand. Vollständiges bidirektionales Sync (alle fremden Ausgaben offline lesen) ist deutlich komplexer und kann in Phase 2 nachgeliefert werden.
07 — KI & Spracheingabe

Ausgaben per Sprache erfassen

„Abendessen 48 Euro, bezahlt von Anna, aufteilen auf alle" — KI versteht und befüllt das Formular automatisch.

✓ Mit allen drei Ansätzen umsetzbar — Capacitor ist ideal
Die Kombination aus nativem Mikrofon-Zugang via Capacitor und der Claude API (Anthropic) oder OpenAI macht dies sehr elegant umsetzbar. Kein MCP nötig — MCP ist ein Protokoll für KI-Agenten-Tools, nicht für Spracheingabe. Für Sprache braucht es: Speech-to-Text + KI-Parsing.

ℹ Was ist MCP?
MCP (Model Context Protocol) ist ein Standard von Anthropic, mit dem KI-Assistenten externe Tools aufrufen können — z.B. Datenbanken lesen, APIs steuern. Für TeileniX wäre ein MCP-Server denkbar, der Claude erlaubt direkt Ausgaben in Supabase zu schreiben. Das ist jedoch eine fortgeschrittene Architektur und für den Nutzer nicht sichtbar anders als die einfachere API-Variante. Empfehlung: Zuerst einfache Claude API nutzen, MCP später als Erweiterung.

PWAs nutzen Service Workers und IndexedDB — das sind Browser-native Mechanismen für Offline-Betrieb.

Technische Bausteine für PWA-Offline
Service Worker — Netz abfangen, aus Cache bedienen next-pwa Plugin
IndexedDB — Pending-Ausgaben lokal speichern idb-keyval (2kb)
Background Sync — Android: automatisch senden wenn online Service Worker API
Network Status — Online/Offline erkennen navigator.onLine
Web App Manifest — Installierbar, Splashscreen, Icon manifest.json

Aufwand PWA-Spracheingabe

Gesamt ~1–2 Tage
✓ PWA-Fazit: Solide Grundlage, mit klaren Grenzen
Eine PWA für TeileniX ist heute schon sehr gut umsetzbar — Offline-Erfassung, KI-Spracheingabe und Installierbarkeit sind alle machbar. Die größte Einschränkung ist iOS: kein Background Sync (Sync erst beim nächsten Öffnen der App) und Web Speech API ist unzuverlässig → daher Whisper-Variante bevorzugen. Android verhält sich deutlich näher an einer nativen App. Empfehlung: PWA als schneller erster Schritt, später mit Capacitor zu nativer App erweitern.
