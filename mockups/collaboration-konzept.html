Was lokal nötig ist — für jeden

Beide Entwickler brauchen eine identische lokale Umgebung. Das sind die Mindestvoraussetzungen für eine produktive Claude-Code-Session mit TeileniX.

Node.js + npm

TeileniX läuft auf Next.js und benötigt Node.js in Version 20 oder neuer. npm ist standardmäßig dabei. nvm (Node Version Manager) empfohlen für einfaches Versionsmanagement.

Versionskontrolle

Git-Workflow für zwei Personen

Mit zwei aktiven Entwicklern ist ein klarer Branch-Workflow wichtig. Direkte Pushes auf main sind verboten — alles läuft über Feature-Branches und Pull Requests.

Die goldene Regel

  • Jedes Feature / jeder Bugfix bekommt einen eigenen Branch
  • Branch-Namen: feature/xyz, fix/abc
  • Fertige Branches werden als Pull Request geöffnet
  • Der jeweils andere Entwickler reviewed und mergt
  • Nach dem Merge: Branch löschen, von main pullen
  • Regelmäßig von main rebeasen, um Konflikte klein zu halten

Was vermieden werden sollte

  • Direkte Commits auf main (auch nicht durch force-push)
  • Lange lebende Feature-Branches (>2–3 Tage)
  • Merge-Commits ohne PR (direkte git merge)
  • .env.local oder andere Secrets committen
  • git push --force auf main

Typischer Arbeitsablauf

# 1. Immer von aktuellem main starten git checkout main git pull origin main # 2. Neuen Feature-Branch erstellen git checkout -b feature/ausgabe-bearbeiten # 3. Claude Code starten und entwickeln claude # 4. Änderungen committen (Claude macht das oft automatisch) git add src/components/expenses/EditExpense.tsx git commit -m "feat: Ausgaben direkt bearbeitbar machen" # 5. Branch pushen und PR öffnen git push -u origin feature/ausgabe-bearbeiten gh pr create --title "feat: Ausgaben bearbeiten" --body "..." # 6. Nach Merge: aufräumen git checkout main && git pull && git branch -d feature/ausgabe-bearbeiten

Branch benennen

feature/ für neue Features, fix/ für Bugfixes, refactor/ für Umstrukturierungen, chore/ für Wartungsarbeiten.

Commit-Nachrichten

Conventional Commits: feat:, fix:, refactor:. Claude Code hält sich standardmäßig daran — einfach die bestehende Konvention fortführen.

Merge-Strategie

Squash-Merge oder Rebase-Merge auf main — so bleibt der Verlauf sauber. Kein regulärer Merge-Commit (vermeidet das "Merged feature-xyz into main"-Rauschen).

Branch-Schutz auf main

Im GitHub-Repository unter Settings → Branches: Branch Protection Rule für main aktivieren. Kein direktes Pushen, mindestens 1 Reviewer für PRs.

Die CLAUDE.md-Datei im Repository-Root wird von Claude Code bei jedem Start automatisch eingelesen. Sie gibt Claude den notwendigen Kontext, damit jede Session nahtlos fortgesetzt werden kann — für beide Entwickler identisch.

Projektbeschreibung

Was ist TeileniX? Für wen? Was ist der aktuelle Stand? Eine präzise Beschreibung in 3–5 Sätzen erspart Claude langes Einlesen und verhindert falsche Annahmen.

# TeileniX — CLAUDE.md ## Was ist TeileniX? TeileniX ist eine mobile-first Reisekostenapp für Familien und Freundesgruppen. Nutzer können Ausgaben erfassen, aufteilen und ausgleichen. Stack: Next.js App Router, Supabase, TypeScript, Tailwind CSS. Aktuell in aktivem Beta-Test.
🏗

Tech Stack & Architektur

Welche Technologien, welche Ordnerstruktur, welche Patterns werden verwendet. So trifft Claude automatisch die richtigen Entscheidungen ohne extra Anweisungen.

## Tech Stack - Next.js 15 App Router (nicht Pages Router!) - Supabase (Auth, PostgreSQL, Realtime, Storage) - TypeScript strict mode - Tailwind CSS v4 - shadcn/ui Komponenten - Zod für Validierung - Supabase SSR für Server Components
📏

Conventions & Regeln

Naming Conventions, Code-Style, Dateistruktur-Regeln. Claude hält sich daran und produziert konsistenten Code, der zu dem passt, was schon existiert.

## Conventions - Komponenten: PascalCase, Dateien: kebab-case - API-Routes: /app/api/[ressource]/route.ts - DB-Typen aus /src/types/database.types.ts - Keine any-Types, immer strikte Typen - Server Actions bevorzugen für Mutationen - Fehlerbehandlung immer mit try/catch + Toast
🚫

Was Claude NICHT tun soll

Explizite Verbote verhindern, dass Claude unerwünschte Änderungen macht. Besonders wichtig für sensible Bereiche wie Auth, Datenbank-Migrationen oder externe Schnittstellen.

## Nicht tun - Keine Supabase-Migrations ohne explizite Anfrage - Keine Änderungen an supabase/migrations/ ohne Absprache - Nicht die Navigation-Struktur ändern (ist festgelegt) - Keine npm-Pakete hinzufügen ohne Rückfrage - CLAUDE.md und Memory-Dateien nicht selbst ändern

Minimale CLAUDE.md — Vorlage für TeileniX

Diese Datei existiert schon oder sollte sofort angelegt werden. Tobi profitiert sofort von einem gut befüllten CLAUDE.md — Claude braucht weniger Erklärung und macht weniger falsche Annahmen.

# TeileniX ## Projektübersicht Mobile-first Reisekostenapp für Gruppen. Next.js 15 App Router, Supabase (Auth + DB + Realtime), TypeScript strict, Tailwind v4, shadcn/ui. ## Ordnerstruktur src/app/(app)/trips/[tripId]/ → Trip-bezogene Seiten src/components/ → Wiederverwendbare Komponenten src/lib/ → Utilities und Supabase-Clients supabase/migrations/ → SQL-Migrations (nicht ohne Absprache ändern!) ## Aktuelle Entwicklung [Hier: Was gerade gebaut wird, offene TODOs, bekannte Bugs] ## Nicht tun - Supabase Migrations ohne explizite Anfrage erstellen - any-Types verwenden - npm-Pakete ohne Rückfrage installieren

Projektgedächtnis

Memory-Strategie — was wohin gehört

Claude Code kann sich Informationen in Memory-Dateien merken. Die Faustregel: Alles, was für das Projekt nützlich ist, gehört ins Git-Repo. Alles, was persönlich oder user-spezifisch ist, bleibt lokal.

Ins Git-Repo (für beide)

  • Technische Architektur-Entscheidungen (warum wurde X gewählt?)
  • Projektübersicht und Tech-Stack (project_overview.md)
  • Bekannte Bugs und offene TODOs
  • Datenbank-Schema-Beschreibungen
  • Deployment-Infos und Server-Konfiguration
  • Wichtige Code-Patterns die im Projekt verwendet werden

Lokal bleiben (nicht ins Git)

  • Persönliches Feedback und Präferenzen
  • User-Memories mit persönlichen Vorlieben
  • Sensible Zugangsdaten oder Keys (nie ins Git!)
  • Lokale Entwicklungsnotizen die nur einen betreffen
  • Temporäre Debugging-Notizen

.gitignore — Memory-Strategie

# .gitignore Ergänzung für Claude Memory # Persönliche Memories lokal halten .claude/memory/user_*.md .claude/memory/feedback_*.md .claude/settings.json # Projekt-Memories ins Git (kein Ignore) # memory/project_overview.md ← ins Git # memory/architecture.md ← ins Git # memory/decisions.md ← ins Git # Niemals ins Git .env.local .env*.local

Der memory/-Ordner im Projekt-Root enthält die geteilten Projektinformationen. Tobis Claude-Installation liest sie genauso wie deine — beide haben identischen Kontext.

Umsetzungsplan

Roadmap: Tobi zum ersten Commit

Schritt für Schritt von null zur ersten gemeinsamen Pull Request — so schnell wie möglich, ohne etwas zu vergessen.

1

GitHub Repository vorbereiten ~10 Min

Repository auf privat stellen (falls noch nicht). Unter Settings → Collaborators Tobis GitHub-Account einladen. Branch-Protection für main aktivieren: kein direktes Pushen, mindestens 1 Review für PRs. Tobi nimmt die Einladung an.

2

Supabase-Zugriff für Tobi ~5 Min

Im Supabase Dashboard unter Organization → Members Tobis E-Mail-Adresse einladen. Rolle: Developer (kann alles lesen und migrieren, aber keine Billing-Änderungen). Tobi nimmt die Einladung an und hat vollen Dashboard-Zugriff.

3

.env.local sicher übermitteln ~5 Min

Die aktuelle .env.local-Datei (mit NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY etc.) über einen sicheren Kanal teilen: 1Password Secure Share, Bitwarden Send oder Signal. Tobi legt die Datei im geklonten Repo-Root ab.

4

Tobis lokale Umgebung einrichten ~30 Min

Node.js v20+ via nvm, Git, VS Code oder Cursor installieren. Repo klonen. Claude Code CLI installieren: npm install -g @anthropic-ai/claude-code. Mit eigenem Anthropic-Konto anmelden. npm install und npm run dev ausführen.

5

CLAUDE.md anlegen / optimieren ~20 Min

CLAUDE.md im Repo-Root anlegen oder prüfen, ob sie vollständig ist. Projektbeschreibung, Tech Stack, Conventions und Verbote eintragen. Committen und pushen — damit liest Tobis Claude sofort denselben Kontext bei der ersten Session.

6

Erster gemeinsamer Commit (Test-PR) ~15 Min

Tobi erstellt einen Feature-Branch (feature/tobi-hello-world), macht eine kleine Änderung (z.B. einen Kommentar oder eine README-Zeile), pusht den Branch und öffnet einen Pull Request. Du reviewst und mergst — der Workflow ist damit einmal vollständig durchgelaufen.

7

Erste echte Aufgabe assignen ~10 Min

Ein konkretes, abgegrenztes Feature oder einen Bugfix aus dem Backlog für Tobi aussuchen. GitHub Issues als Aufgaben-Tracker nutzen. Tobi startet seine erste echte Claude-Code-Session mit dem Projekt-Kontext aus CLAUDE.md und den Memory-Dateien.

Tobi macht seinen ersten Commit

Branch anlegen → Claude starten → Feature bauen → PR öffnen → Review → Merge. Dieser Ablauf ist das Herz der Zusammenarbeit.

Wer mergt PRs?

Grundsatz: Nie den eigenen PR mergen. Der jeweils andere Entwickler reviewed und mergt — sorgt für vier Augen und bessere Codequalität.

Konflikte auflösen

Bei Merge-Konflikten: lokal von main rebeasen (git rebase main), Konflikte manuell lösen, dann neu pushen. Claude Code kann dabei helfen.

Kommunikation

PR-Beschreibungen ausführlich halten — was wurde geändert, warum, was wurde getestet. Das schafft Transparenz und hilft Claude beim nächsten Mal.
