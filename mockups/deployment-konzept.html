Ausgangslage

Der Testkontext

TeileniX ist eine mobile-first Reisekostenapp (Next.js App Router, Supabase, TypeScript, Tailwind). Ziel ist ein realer Gruppentest unter Alltagsbedingungen — spontane Eingaben, schlechtes Netz, gemischte Geräte.

Anforderungen

  • Komfort wie eine native App (smooth, snappy)
  • Funktioniert auch bei schlechtem Mobilfunk
  • Kein App-Store-Download nötig (für schnellen Test-Start)
  • Echtzeit-Synchronisation zwischen allen 16 Personen
  • Kein separates Backend — Supabase bleibt bestehen
  • Minimaler Entwicklungsaufwand vor dem Gruppentest

Herausforderungen

  • iOS Safari beschränkt PWA-Features (Push, Hintergrundprozesse)
  • 16 Supabase Realtime-Verbindungen gleichzeitig
  • Netzausfall beim Hinzufügen von Ausgaben im Urlaub
  • Unterschiedliche Gerätegeschwindigkeiten in der Gruppe
  • Onboarding ohne Tech-Kenntnisse (Großeltern, Kinder)

Detailanalyse

Jede Variante im Detail

Variante A — Optimiertes PWA

Empfohlen für Gruppentest 8–20h

Die bestehende Next.js App erhält ein vollständiges Web App Manifest, einen Service Worker (z.B. via next-pwa oder Workbox) und einen Install-Banner. Nutzer können die App auf dem Homescreen installieren — sie startet vollbild ohne Browser-Chrome.

✦ Vorteile

  • Keine Codebase-Änderung nötig — reines Konfiguration
  • Homescreen-Icon auf iOS und Android
  • Vollbild-Modus ohne Browser-Leiste
  • Offline-Cache für bereits geladene Seiten
  • Sofort testbereit — Link teilen, fertig
  • Kein App Store Review-Prozess

✦ Einschränkungen

  • iOS: Push Notifications nur ab iOS 16.4 (und nur wenn installiert)
  • iOS: Kein Background Sync, kein Vibrations-API
  • Kein echter Offline-First (nur Cache-First für Assets)
  • Installation muss manuell erklärt werden (Add to Homescreen)
  • Safari PWA-Support bleibt hinter Chrome/Android zurück
// next.config.js — PWA mit next-pwa aktivieren const withPWA = require('next-pwa')({ dest: 'public', register: true, skipWaiting: true, runtimeCaching: workboxCaching, // Cache-Strategie buildExcludes: [/middleware-manifest\.json$/], });

Variante B — Capacitor-Wrapper

Upgrade-Pfad 40–80h

Capacitor.js (von Ionic) verpackt die bestehende Web-App in einen nativen iOS/Android Container. Die App wird im App Store vertrieben, hat Zugang zu nativen APIs (Kamera, Haptik, Push via APNs/FCM) und fühlt sich echter an.

✦ Vorteile

  • 95% des bestehenden Next.js-Codes bleibt unverändert
  • Echter App Store Vertrieb (professioneller Eindruck)
  • Voller Zugang zu nativen iOS/Android APIs
  • Haptisches Feedback, Push Notifications, Camera
  • Kein Browser-Chrome — echte native Shell
  • Capacitor hat exzellente Next.js-Kompatibilität

✦ Einschränkungen

  • Xcode + Android Studio Setup nötig (macOS für iOS)
  • Apple Developer Account: 99$/Jahr
  • App Store Review: 1–3 Tage für initiale Einreichung
  • Next.js SSR muss auf Static Export umgestellt werden
  • Supabase Auth-Flow muss für native Deeplinks angepasst werden

Variante C — Expo / React Native

Langfristige Option 200–400h

Eine vollständige Neuentwicklung als echte React Native App mit Expo. Maximale native Performance, perfekte Plattform-Integration — aber komplette Neuentwicklung aller UI-Komponenten.

✦ Vorteile

  • Echte native Performance — keine Web-Layer
  • Native Gesten, Animationen, Scrolling
  • Volle Plattform-Integration auf beiden OS
  • Expo EAS Build vereinfacht Deployment massiv
  • Supabase JS SDK funktioniert identisch in RN

✦ Einschränkungen

  • Komplette Neuentwicklung — kein Tailwind, keine Web-Komponenten
  • Sehr hoher Zeitaufwand vor dem Gruppentest
  • Separate Codebasis zu pflegen (Web + Native)
  • React Native-Spezifische Debugging-Komplexität
  • Für einen ersten Gruppentest deutlich überdimensioniert

Variante D — Vercel Edge + PWA Optimierung

Empfohlen für Gruppentest 10–20h

Die App wird auf Vercel deployed (oder bleibt dort) mit aktiviertem Edge Network. Supabase Realtime wird für Live-Synchronisation aller 16 Nutzer eingerichtet. Optimistic UI sorgt dafür, dass Aktionen sofort reagieren — auch bei 200ms Netzlatenz.

✦ Vorteile

  • TTFB unter 50ms durch Edge-nahe Auslieferung
  • Optimistic UI: UI reagiert sofort, Sync im Hintergrund
  • Supabase Realtime: Änderungen aller 16 User sofort sichtbar
  • Zero Deployment-Aufwand (Vercel CI/CD already works)
  • Kein App Store nötig — URL teilen reicht für Test
  • Verbessert die "native-feel" deutlich ohne nativen Code

✦ Einschränkungen

  • Gleiche iOS Safari PWA-Limitierungen wie Variante A
  • Optimistic UI benötigt sorgfältige Fehlerbehandlung (Rollback)
  • Supabase Free Tier: max. 200 gleichzeitige Realtime-Verbindungen (reicht)
  • Edge-Caching muss mit Supabase RLS-Auth koordiniert werden

Supabase Realtime

Alle 16 Personen sehen Ausgaben und Packlisten-Updates sofort — ohne manuelle Aktualisierung. WebSocket-basiert, direkt in Supabase integriert.

Offline-First Strategien

Im Urlaub ist das Netz oft schlecht. Mit einer Offline-Queue werden Eingaben lokal gespeichert und bei Netzwiederkehr automatisch synchronisiert.

Die optimale Strategie

Variante A + D kombinieren

Für den Gruppentest in der nächsten Zeit ist die Kombination aus PWA-Optimierung (A) und Vercel Edge + Supabase Realtime (D) die klar beste Wahl. Geringer Aufwand, maximale Wirkung, keine neue Infrastruktur.

Bereit in 2–3 Tagen ~20–30h Entwicklung 100% bestehender Code Kein App Store nötig

Was konkret gebaut wird

  • Web App Manifest mit Icon (alle Größen) + Splashscreen
  • Service Worker für Asset-Caching (next-pwa)
  • Install-Banner mit iOS-Anleitung in der App
  • Supabase Realtime auf Ausgaben + Packliste
  • Optimistic UI für Ausgabe-Hinzufügen
  • Offline-Queue für Eingaben ohne Netz
  • Vercel Edge Config für schnellste Auslieferung

Ergebnis für die Gruppe

  • App öffnet in <1 Sekunde auf allen Geräten
  • Ausgaben erscheinen bei allen 16 sofort
  • Funktioniert auch bei schlechtem Netz im Urlaub
  • Homescreen-Icon — fühlt sich wie echte App an
  • Kein Download, kein App Store — einfach Link schicken
  • Großeltern-tauglich durch einfaches Onboarding

Upgrade-Pfad: Variante B (Capacitor)

Nach dem Gruppentest

Falls nach dem Gruppentest eine echte App-Store-Distribution gewünscht wird (professionelleres Auftreten, Push-Notifications für alle iOS-Versionen, besseres Installations-Erlebnis): Capacitor.js ist der natürliche nächste Schritt. 95% des Codes bleibt unverändert, der Aufwand ist überschaubar, und es entsteht eine vollwertige App im App Store.

Umsetzungsplan

Roadmap zum Gruppentest

Konkrete Schritte für die Umsetzung von Variante A+D — von der Entwicklung bis zum ersten Testabend mit der Gruppe.

1

Vercel + Supabase Realtime konfigurieren ~4h

Supabase Realtime auf den relevanten Tabellen aktivieren (expenses, packlist_items). Vercel Production-Domain einrichten, Environment Variables prüfen. Edge Network ist bei Vercel Pro automatisch aktiv.

2

Optimistic UI für Ausgaben implementieren ~6h

useOptimistic Hook in den Ausgaben-Formularen einsetzen. Rollback-Logik bei Fehlern. Toast-Benachrichtigungen für Fehler. Supabase Realtime-Listener für automatische Updates registrieren.

3

PWA Manifest + Service Worker ~6h

next-pwa installieren und konfigurieren. App-Icons in allen nötigen Größen erstellen (512px, 192px, 180px für iOS). Splashscreen für iPhone. manifest.json mit korrekter display: standalone konfiguration.

4

Install-Banner + iOS-Onboarding ~4h

Install-Banner für Android (BeforeInstallPrompt). iOS-spezifische Anleitung mit Screenshot-Anleitung beim ersten Besuch (einmalig zeigen, dann localStorage-Flag). Android zeigt automatischen Prompt.

5

Offline-Queue implementieren (optional) ~8h

IndexedDB-basierte Queue für Aktionen bei fehlender Verbindung. Background Sync via Service Worker für automatisches Abspielen. Visuelle Anzeige für "ausstehende" Einträge in der UI.

6

Gruppentest-Vorbereitung ~2h

Einladungs-URL vorbereiten. Kurze Anleitung (Whatsapp-Nachricht oder PDF) für nicht-technische Nutzer: "Öffne diesen Link in Safari und füge die App zum Homescreen hinzu." Test-Reise anlegen mit allen 16 Personen.

Wann welche Variante wählen?

Kurze Übersicht für verschiedene Szenarien und Anforderungen.

Gruppentest in <1 Woche

→ Variante A + D: PWA + Vercel Edge. Schnellste Umsetzung, sofort teilbar, kein App Store.

App Store Distribution gewünscht

→ Variante B: Capacitor-Wrapper nach dem ersten Test. Apple Developer Account notwendig.

Maximale native Performance

→ Variante C: Expo/React Native — aber erst nach echtem Produkt-Markt-Fit und größerem Budget.

Schlechte Netzverbindung im Urlaub

→ Variante A + D mit Offline-Queue: Eingaben werden lokal gespeichert und bei Netz synchronisiert.
