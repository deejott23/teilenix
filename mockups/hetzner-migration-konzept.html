Supabase Cloud vs. Self-Hosted

Supabase lässt sich theoretisch selbst hosten — aber der Aufwand ist erheblich. Hier ein ehrlicher Vergleich zwischen dem bequemen Cloud-Service und dem vollständig selbstgehosteten Stack.

Bereich Supabase Cloud (behalten) Self-Hosted auf Hetzner
Auth (Email, Magic Link) Sofort, konfigurierbar Eigener GoTrue-Stack nötig
Realtime (WebSockets) Integriert, skaliert auto. Eigener Realtime-Server, komplex
Storage S3-kompatibel, sofort nutzbar MinIO oder S3-Anbindung nötig
Row Level Security (RLS) Voll unterstützt Voll unterstützt (PostgreSQL-Feature)
Backups Automatisch täglich (Pro) Selbst einrichten (pg_dump, Cron)
Updates / Sicherheitspatches Automatisch Manuell (Docker pull, Downtime)
PostgreSQL-Version Managed, immer aktuell Frei wählbar, selbst updaten
Dashboard / Studio Vollständig, poliert Supabase Studio lokal (Docker)
Kosten / Monat Free Tier: €0 (bis 50k Rows) Server-Kosten + Overhead ~€10–15 extra
Migrations (CLI) Vollständig unterstützt Vollständig unterstützt
Setup-Aufwand Zero 40–80h initial

Achtung: Self-Hosted Supabase ist aufwendig

Supabase Self-Hosted besteht aus über 10 Docker-Containern (PostgreSQL, GoTrue, PostgREST, Realtime, Storage, Kong API Gateway, Studio u.a.). Das Einrichten, Warten und Updaten dieses Stacks ist eine eigene Arbeit — und bei einem kleinen Projekt wie TeileniX ist der Aufwand klar nicht gerechtfertigt. Supabase Cloud auf dem Free Tier zu behalten ist fast immer die bessere Entscheidung.

Technische Umsetzung

Was für Variante A nötig ist

Die konkrete Infrastruktur für Variante A — Next.js auf Hetzner mit Docker, Caddy als Reverse Proxy und GitHub Actions für automatisches Deployment.

🐳

Docker + Docker Compose

Next.js läuft als Docker-Container auf dem Hetzner-Server. Docker Compose verwaltet den Container mit dem richtigen Restart-Verhalten und Port-Mapping. Kein PM2 nötig.

# docker-compose.yml services: nextjs: build: . restart: always ports: - "3000:3000" env_file: - .env.production environment: - NODE_ENV=production
Restart on Crash Kein PM2 nötig
🔐

Caddy als Reverse Proxy

Caddy übernimmt SSL-Terminierung (Let's Encrypt automatisch), leitet Requests von Port 443 an den Next.js-Container weiter und handhabt HTTP→HTTPS Redirects ohne Konfiguration.

# Caddyfile teilenix.de { reverse_proxy localhost:3000 # SSL: automatisch via Let's Encrypt # HTTP → HTTPS: automatisch }
SSL automatisch Zero-Config HTTPS
⚙️

GitHub Actions — CI/CD

Bei jedem Push auf main baut GitHub Actions ein neues Docker-Image und deployt es auf den Hetzner-Server via SSH. Kein manuelles Deployment, zero Downtime via Docker rollout.

# .github/workflows/deploy.yml on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Deploy to Hetzner run: | ssh deploy@$SERVER_IP \ "cd /app && git pull && docker compose up -d --build"
Push → Deploy SSH-basiert
🔑

Environment Variables

Die Supabase-Keys und andere Secrets werden als .env.production-Datei direkt auf dem Server abgelegt — nicht im Git-Repo. GitHub Actions liest Server-SSH-Keys aus GitHub Secrets.

# Auf dem Hetzner-Server einmalig anlegen cat > /app/.env.production << 'EOF' NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbG... SUPABASE_SERVICE_ROLE_KEY=eyJhbG... EOF # Berechtigungen einschränken chmod 600 /app/.env.production
Nie im Git! Direkt auf Server
🌐

Domain & DNS

Der DNS-A-Record der Domain zeigt nach der Migration auf die Hetzner-IP statt auf Vercel. Propagation dauert je nach TTL 5 Minuten bis 24 Stunden. Zuerst alles auf Hetzner testen, dann DNS umziehen.

# DNS-Einträge beim Domainprovider @ A 65.21.xxx.xxx # Hetzner IP www CNAME teilenix.de. # Weiterleitung # IP des Hetzner-Servers herausfinden hcloud server describe mein-server --output json \ | jq '.public_net.ipv4.ip'
TTL niedrig setzen Erst testen, dann umziehen
📁

Dockerfile für Next.js

Ein optimiertes Dockerfile mit Multi-Stage Build: Erst Build-Stage (installiert alle Dependencies, baut die App), dann Production-Stage (nur Runtime-Dependencies). Ergebnis: kleines, schnelles Image.

# Dockerfile (Multi-Stage) FROM node:20-alpine AS builder WORKDIR /app COPY package*.json . RUN npm ci COPY . . RUN npm run build FROM node:20-alpine AS runner WORKDIR /app COPY --from=builder /app/.next/standalone . EXPOSE 3000 CMD ["node", "server.js"]
Multi-Stage = klein output: standalone

Einschränkungen & Risiken

Was man wissen muss

Hetzner ist keine Drop-in-Replacement für Vercel. Diese Einschränkungen sind real und sollten vor der Migration bewusst abgewogen werden.

Kein Edge Network

Performance

Vercel liefert Inhalte von über 70 Edge-Standorten weltweit aus — der TTFB liegt bei 50–100ms. Hetzner hat einen Standort (Frankfurt oder Nürnberg). Für europäische Nutzer ist der Unterschied minimal, für andere Kontinente deutlicher. Für TeileniX — eine App hauptsächlich für deutsche Familien — ist das praktisch irrelevant.

Auswirkung in der Praxis

  • TTFB steigt von ~80ms auf ~150ms für deutsche Nutzer
  • Internationale Nutzer (USA, Asien) spüren mehr Latenz
  • Static Assets (JS, CSS) sollten über ein CDN (Cloudflare) ausgeliefert werden

Milderung

  • Cloudflare Proxy vor Hetzner (kostenlos) gibt CDN für Static Assets
  • Next.js output: standalone baut optimierten Server
  • Für TeileniX-Nutzerschaft (DE/AT/CH): kein relevanter Unterschied

Keine automatischen Preview Deployments

Developer Experience

Bei Vercel bekommt jeder Pull Request automatisch eine Preview-URL. Das ist enorm praktisch für Reviews. Auf Hetzner muss das selbst gebaut werden — z.B. über einen staging-Branch, der auf einen zweiten Docker-Container deployed wird.

Was fehlt

  • Kein automatisches Preview-Deployment für jeden PR
  • Kein Vercel-Kommentar im PR mit der Preview-URL
  • Reviewer müssen Code lokal auschecken zum Testen

Alternativen

  • Staging-Environment auf demselben Server (Docker, anderer Port)
  • GitHub Actions deployt staging-Branch automatisch
  • Für 2-Personen-Team meist ausreichend

Wartungsaufwand

Operations

Ein selbstverwalteter Server braucht gelegentliche Aufmerksamkeit: OS-Updates, Docker-Updates, Monitoring auf Ressourcenverbrauch. Das ist bei einem kleinen Projekt überschaubar — aber es ist nicht null.

Aufgaben

  • Monatlich: apt upgrade auf dem Server
  • Docker-Images aktuell halten (Security-Patches)
  • Festplattennutzung und RAM überwachen
  • Backup-Strategie für den Server (Hetzner Snapshots: ~€0,01/GB)

Aufwand in der Praxis

  • Realistisch: ~15–30 Minuten pro Monat
  • Hetzner bietet automatische OS-Updates (unattended-upgrades)
  • Docker Watchtower kann Container automatisch updaten
  • Hetzner Backups: täglich automatisch konfigurierbar
Umsetzungsplan

Roadmap: Von Vercel zu Hetzner

Schritt für Schritt zur Migration — zuerst alles auf Hetzner zum Laufen bringen, dann erst DNS umziehen. Kein Downtime-Risiko durch diesen Ansatz.

1

Hetzner Server einrichten ~30 Min

Server im Hetzner Cloud Dashboard auswählen (CX21 empfohlen: 2 vCPU, 4 GB RAM, Ubuntu 22.04). SSH-Key hinterlegen. Server erstellen. Erstes Login via SSH, System updaten: apt update && apt upgrade -y. Deploy-User anlegen.

2

Docker + Docker Compose installieren ~15 Min

Docker über das offizielle Skript installieren: curl -fsSL https://get.docker.com | sh. Docker Compose v2 ist ab Docker 23 inklusive. Deploy-User zur docker-Gruppe hinzufügen. Docker-Daemon testen mit docker run hello-world.

3

Caddy als Reverse Proxy einrichten ~20 Min

Caddy als Docker-Container (oder apt-Paket) installieren. Caddyfile mit Domain und Reverse-Proxy-Regel anlegen. Ports 80 und 443 in Hetzner Firewall öffnen. Caddy starten — SSL-Zertifikat wird automatisch via Let's Encrypt angefragt.

4

Dockerfile + docker-compose.yml erstellen ~1h

Multi-Stage Dockerfile für Next.js im Repo anlegen. output: 'standalone' in next.config.ts aktivieren (erzeugt minimales Server-Bundle). docker-compose.yml mit korrekten Port-Mappings und env_file. Lokal testen: docker compose up --build.

5

GitHub Actions Workflow anlegen ~2h

Workflow-Datei .github/workflows/deploy.yml erstellen. SSH-Private-Key als GitHub Secret hinterlegen. Workflow: checkout → build Docker Image → SSH zum Server → pull + rebuild. Ersten manuellen Deploy-Test durchführen.

6

Environment Variables auf Server konfigurieren ~15 Min

.env.production-Datei auf dem Server anlegen (nie im Git!). Alle Supabase-Keys aus der lokalen .env.local kopieren. Docker Compose referenziert die Datei via env_file. Berechtigungen auf 600 setzen. App neu starten und testen.

7

App auf Hetzner testen (vor DNS-Umzug) ~1h

App über die Hetzner-IP oder einen Testdomain-A-Record testen. Auth-Flow mit Supabase prüfen (Redirect-URLs müssen ggf. in Supabase erweitert werden). Realtime-Funktionen testen. Performance checken. Erst wenn alles funktioniert: DNS umziehen.

8

DNS auf Hetzner umziehen ~5 Min

TTL beim Domainprovider auf 300 Sekunden (5 Minuten) reduzieren — mindestens 24h vorher. A-Record von Vercel-IP auf Hetzner-IP ändern. Propagation abwarten. App über die echte Domain testen. Vercel-Deployment kann danach abgekündigt werden.

Wann Variante A wählen

Wenn das Ziel primär Kostensenkung ist, Supabase reibungslos funktioniert und kein massiver Skalierungsbedarf besteht. Beste Wahl für TeileniX in der aktuellen Phase.

Wann Variante B erwägen

Wenn vollständige Datenkontrolle (DSGVO, Unternehmensrichtlinien) nötig ist oder Supabase zu teuer wird (Pro Plan bei wachsenden Datenmengen). Für ein privates Hobbyprojekt nicht empfohlen.

Wann bei Vercel bleiben

Wenn Preview Deployments täglich genutzt werden, Edge-Performance kritisch ist oder die Entwicklungszeit wichtiger ist als die $20/Monat. Vercel ist objektiv bequemer.

Rollback-Strategie

Vercel-Deployment nicht sofort löschen — erst 2–4 Wochen auf Hetzner beobachten. Bei Problemen einfach DNS zurückzeigen lassen. Supabase ist unverändert, kein Datenverlustrisiko.
