Wie ermittle ich die richtigen Testfälle?

Vier Quellen kombinieren — dann sind alle kritischen Pfade abgedeckt.

🗺️

A) User Journey Mapping

Jeden Weg, den ein Nutzer durch die App nehmen kann, als Test abbilden.

  • → Registrieren / Einloggen
  • → Reise erstellen, beitreten
  • → Ausgabe anlegen (alle Split-Modi)
  • → Abrechnung anzeigen & exportieren
  • → Reise beenden
💥

B) Fehlerfälle & Edge Cases

Was passiert wenn etwas schiefläuft — das sind oft die unentdeckten Bugs.

  • → Betrag = 0 oder negativ
  • → Kein Teilnehmer ausgewählt
  • → Einladungscode ungültig
  • → Offline-Modus: Ausgabe speichern
  • → Gleichzeitiges Bearbeiten
🔐

C) Sicherheits- & Berechtigungstests

Supabase RLS muss aus Nutzerperspektive geprüft werden.

  • → Nutzer sieht nur eigene Reisen
  • → Gast kann keine Reise beenden
  • → Unauthentifiziert → Redirect /login
  • → Fremde Ausgabe nicht editierbar
📊

D) Regressionsbasiert (Bug-History)

Jeden gefixten Bug als Test festhalten, damit er nicht wiederkommt.

  • → Gesperrtes Eingabefeld (SplitOverrideEditor)
  • → Falsche Aufteilung bei Multi-Payer
  • → Ausgaben-Summen-Diskrepanz

Automatische Testgenerierung

Drei Wege, Testcode ohne manuelles Schreiben zu erzeugen — von einfach bis fortgeschritten.

🎥 Playwright Codegen

App im Browser bedienen → Playwright zeichnet automatisch TypeScript-Testcode auf.

npx playwright codegen
  http://localhost:3000

🤖 Claude Code AI

Claude analysiert Routen, Komponenten und Zod-Schemas und generiert vollständige Testsuiten.

# Prompt-Beispiel:
"Generiere Playwright-Tests
für ExpenseForm.tsx"

📸 Visual Snapshots

Playwright macht Screenshots — bei Render-Änderungen schlägt der Test automatisch an.

expect(page)
  .toHaveScreenshot(
    'expense-form.png');

Claude Code als Testgenerator — Workflow

1

Komponente zeigen

Claude öffnet die relevante Komponente (z.B. ExpenseForm.tsx) und liest den Zod-Schema für Validierungsregeln.

2

Prompt formulieren

Beispiel: "Erstelle Playwright-Tests für alle Split-Modi in ExpenseForm. Happy Path + ungültige Eingaben. Nutze die bestehende auth-fixture."

3

Generierter Test → Review

Claude generiert die .spec.ts-Datei. Kurzes Review: Selektoren korrekt? Assertions sinnvoll? Dann direkt committen.

4

Test ausführen & verfeinern

npx playwright test --ui — fehlschlagende Tests an Claude zurückgeben: "Test X schlägt fehl, hier der Fehler."

Auth-Problem lösen: Supabase in Tests

Google OAuth lässt sich nicht automatisieren. Zwei bewährte Lösungen:

Option A: Supabase Magic Link (empfohlen)

Einen Test-User anlegen, der per E-Mail-Magic-Link einloggt — kein OAuth-Provider nötig.

// e2e/fixtures/auth.ts
export async function loginAsTestUser(page) {
  const { data } = await supabaseAdmin
    .auth.signInWithPassword({
      email: 'test@teilenix.dev',
      password: process.env.TEST_USER_PW
    });
  // Cookie in Playwright setzen
  await page.context().addCookies([...]);
}

Option B: Lokale Supabase-Instanz

Supabase lokal starten — komplett isoliert, Auth ohne echten Provider möglich.

npx supabase start    # Docker nötig
# → lokale DB mit Seed-Daten
# → auth.users direkt befüllbar
# → RLS testbar ohne Cloud-Kosten

# Seed-Script:
npx supabase db seed

Testdaten-Strategie

🌱 Seed-Fixtures

Eine fixe Testumgebung: 2 User, 1 Reise, 3 Ausgaben → immer gleicher Ausgangszustand.

🏭 Factory Functions

Hilfsfunktionen die Testdaten programmatisch erzeugen und nach dem Test aufräumen (afterEach).

🧹 Cleanup

Nach jedem Test alle erzeugten Datenbankeinträge löschen — Tests bleiben unabhängig voneinander.

Beispiel-Test: Ausgabe erstellen

// e2e/expenses/create-proportional.spec.ts
import { test, expect } from '@playwright/test';
import { loginAsTestUser, seedTrip } from '../fixtures';

test.describe('Ausgabe erstellen – proportionaler Split', () => {
  let tripId: string;

  test.beforeEach(async ({ page }) => {
    await loginAsTestUser(page);
    tripId = await seedTrip({ participants: ['alice', 'bob', 'charlie'] });
    await page.goto(`/trips/${tripId}/expenses/new`);
  });

  test('Happy Path: Ausgabe mit 3 Teilnehmern', async ({ page }) => {
    await page.getByLabel('Titel').fill('Abendessen');
    await page.getByLabel('Betrag').fill('90');
    await page.getByRole('button', { name: 'Speichern' }).click();

    await expect(page.getByText('Abendessen')).toBeVisible();
    await expect(page.getByText('90,00 €')).toBeVisible();
  });

  test('Validierung: leerer Titel → Fehler', async ({ page }) => {
    await page.getByLabel('Betrag').fill('50');
    await page.getByRole('button', { name: 'Speichern' }).click();

    await expect(page.getByText(/Pflichtfeld|required/i)).toBeVisible();
  });

  test('Abrechnungssumme stimmt nach Speichern', async ({ page }) => {
    // Ausgabe €90, 3 Teilnehmer gleiche Anteile → je 30€
    await page.goto(`/trips/${tripId}/settlement`);
    await expect(page.getByText('30,00 €')).toBeVisible();
  });
});

CI/CD Integration

Tests automatisch bei jedem Push ausführen — Fehler vor dem Deployment entdecken.

git push
→
GitHub Action
→
npm run dev + supabase start
→
playwright test
→
Report + Vercel Deploy
# .github/workflows/e2e.yml
name: E2E Tests
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: supabase/setup-cli@v1
      - run: supabase start --ignore-health-check
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test
        env:
          NEXT_PUBLIC_SUPABASE_URL: http://localhost:54321
          TEST_USER_PW: ${{ secrets.TEST_USER_PW }}
      - uses: actions/upload-artifact@v4
        if: failure()
        with:
          name: playwright-report
          path: playwright-report/

✅ Bei Erfolg

Vercel-Deployment läuft durch. Grüner Check im PR.

❌ Bei Fehler

PR wird geblockt. Playwright-Report mit Screenshots und Video als Artifact verfügbar.

Claude Code im Test-Workflow

Claude Code kann in jeder Phase des Testprozesses unterstützen — nicht nur beim Schreiben.

📝 Tests schreiben

Komponenten-Datei zeigen → Claude generiert vollständige Spec-Datei mit Happy Path + Edge Cases.

"Erstelle e2e/trips/create-trip.spec.ts
basierend auf /app/trips/new/page.tsx
und dem TripForm-Schema in types/app.ts"

🔍 Fehler debuggen

Failing Test + Fehlermeldung an Claude → erklärt warum und liefert Fix.

"Test schlägt fehl: Element
getByLabel('Betrag') not found.
Hier die Seite: ExpenseForm.tsx"

♻️ Tests refactoren

Nach UI-Änderungen: veraltete Selektoren automatisch aktualisieren lassen.

"Button-Label hat sich von 'Speichern'
zu 'Ausgabe erfassen' geändert.
Aktualisiere alle Specs."

🗺️ Coverage-Analyse

Claude analysiert bestehende Tests und findet ungetestetenRouten/Komponenten.

"Welche Routen aus /app haben noch
keinen Test in /e2e? Priorisiere nach
Kritikalität für den Nutzer."

Quick Start — In 3 Schritten loslegen

1

Playwright einrichten (30 Min.)

Playwright installieren, Config anlegen, lokale Supabase starten, einen ersten Smoke-Test schreiben der nur prüft: Startseite lädt, Login-Button sichtbar.

npm init playwright@latest
npx supabase init  # falls noch nicht
npx supabase start
2

Auth-Fixture + ersten echten Test (2–3 Std.)

Test-User in Supabase anlegen. Auth-Fixture schreiben. Den wichtigsten Flow testen: Ausgabe erstellen → Abrechnung prüfen. Mit Claude Code generieren lassen.

3

GitHub Actions CI aktivieren (1 Std.)

YAML-Workflow anlegen. Supabase CLI in CI einbinden. Secrets für Test-Credentials hinterlegen. Ab jetzt läuft jeder Push durch die Tests.

Empfohlene Reihenfolge der Testfälle

Nicht alles auf einmal — mit dem Wichtigsten anfangen:

  1. Smoke-Test: App startet, Login-Seite erreichbar
  2. Ausgabe erstellen (proportional, 2 Personen)
  3. Abrechnungsbetrag korrekt nach Ausgabe
  4. Reise erstellen & via Code beitreten
  5. Custom Split testen
  6. Unauthorized-Redirect
  7. PDF-Export (Datei-Download prüfen)
  8. Offline-Queue (Netzwerk deaktivieren, Ausgabe speichern)
